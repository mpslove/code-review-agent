### Architecture Review

🟠 **[HIGH]** `src/summarizer.py:1` — LLM调用职责重复，违反单一职责原则

> Summarizer类中的_llm_merge()方法直接使用requests调用LLM API，与src/reviewer/base.py中的_call_llm()功能重复。这导致LLM调用逻辑分散在两处，维护成本高，且summarizer职责过重（本应仅负责格式化和汇总）。

<details>
<summary>💡 Suggestion</summary>

```
提取公共LLM调用模块（如src/llm.py），由BaseReviewer和Summarizer共同依赖。或者让Summarizer接收一个callable进行LLM调用，实现依赖注入。
```
</details>

🟡 **[MEDIUM]** `src/github.py:73` — post_comment实现冗余且错误处理不一致

> post_comment方法先调用_gh（可能因FileNotFoundError抛出RuntimeError），捕获后再次使用subprocess.run。_gh已经封装了命令执行，这里重复实现，且错误处理路径不统一（一次抛出RuntimeError，一次返回False）。破坏了封装性和一致性。

<details>
<summary>💡 Suggestion</summary>

```
修改_gh函数，使其支持stdin输入（通过stdin参数），然后post_comment统一使用_gh。如果gh不存在，_gh已抛出RuntimeError，post_comment只需捕获并返回False。
```
</details>

🔵 **[LOW]** `tests/benchmark/runner.py:90` — run_all接口设计不合理，隐式依赖外部函数签名

> run_all方法接收review_fn参数，期望签名为(case_name, diff_text) -> list[AgentReviewResult]。但该签名仅在内部文档中约定，没有通过类型注解或抽象类明确。且review_fn的第一个参数case_name在某些场景下可能多余，导致接口灵活性不足。

<details>
<summary>💡 Suggestion</summary>

```
将review_fn定义为Callable[[str, str], list[AgentReviewResult]]并在类型注解中明确。或者改为接受一个实现了特定接口的对象（如ReviewRunner协议），增强可测试性。
```
</details>
